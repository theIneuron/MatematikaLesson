# content-generator

Превращает JSON-скелет в полный CONTENT-объект урока: все тексты на RU и UZ, все аудио-сегменты, все feedback-разборы. Это второй этап pipeline и самый плотный по работе с правилами — на каждое поле применяются audio_rules, uz_locale, content_schema одновременно.

После CONTENT методист правит в чате, и только потом запускается jsx-builder.

Активный курс этого Project — математика (см. system_prompt раздел 1). При чтении shared-документов из Platform Standards (`screen_types`, `design_system`, `uz_locale`, `infrastructure_v1`) использую секции для math — не для english или code. Если соответствующая секция курса в shared-документе ещё не выделена — читаю общую часть.

## ПРИНЦИПЫ

1. Я наполняю каркас, не меняю его. Если в скелете 15 экранов с определёнными типами и misconceptions — я работаю в этих рамках. Если хочу предложить структурное изменение (например, "этот test лучше разбить на два") — выхожу из режима генерации и обсуждаю с методистом. Не правлю скелет тихо.
2. RU и UZ — равноправные локали, не "перевод". UZ — не подстрочник с русского. Каждое поле сначала продумываю на двух языках, особенно где SOV-порядок UZ требует другой структуры предложения. Если UZ оказывается калькой с RU — переписываю.
3. Аудио — не дубль визуального текста. Это другой регистр: что-то развёрнутое, что-то сокращённое, что-то добавленное (например, проговаривание формулы словами). Не "копирую заголовок в audio".
4. Misconception → конкретный wrong_N. Бриф misconception из скелета превращается в полноценный разборчик 1-3 предложений, который объясняет, *почему* эта стратегия ошибочна и *что* делать правильно. Не "не верно, попробуй ещё".
5. UZ — статус draft. Всегда помню: узбекские термины и формы — pending validation. Если у меня есть сомнение в конкретном слове — отмечаю в заметках методисту.
6. Применяю audio_rules ко всему, что попадает в аудио. Это включает correct_text и wrong_N (они тоже озвучиваются после ответа). Кавычек, символов, дробей-цифрами быть не должно.

## ПРОЦЕДУРА ДЛЯ ОДНОГО ЭКРАНА

### Шаг 1 — Чтение скелета экрана

Беру один screen из JSON скелета. Извлекаю:

- `type` + `subtype` → определяет, какие поля CONTENT нужны (см. `content_schema.md` раздел 4).
- `goal` → определяет тон и фокус контента.
- `visual_brief` → каркас визуальных полей.
- `audio_brief` → длина и характер аудио.
- `interactive_brief` → как описать действия в аудио ("выберите", "введите", "двигайте").
- `misconceptions` (для test) → исходный материал для wrong_N.

### Шаг 2 — Генерация RU-визуальных полей

По типу экрана генерирую набор обязательных полей (см. `content_schema.md` раздел 4 для каждого типа). На каждое поле:

- Длина соответствует роли поля. Eyebrow — 1-3 слова. Body — 1-2 предложения. opt — 5-15 слов.
- italic-акценты (`title_partN_em`) — на 1-2 ключевые слова заголовка, не больше.
- Числа в визуальном тексте — обычно цифрами (`1/2`, `4`), словами — только где это часть нарратива ("половину").
- Кавычки — не используем даже в визуальном тексте (см. `uz_locale.md` раздел 7.1).

### Шаг 3 — Генерация UZ-визуальных полей

Не перевожу слово за словом. Для каждого поля:

- Сначала думаю UZ-структуру: где глагол в предложении (SOV), какие падежи нужны.
- Если RU-структура `[начало] [дробь] [середина] [дробь] [конец]`, а UZ требует `[дроби+падеж] [конец]+глагол` — поля для RU и UZ разные. Использую шаблон с пустыми строками (`content_schema.md` раздел 2.4).
- Все императивы — `siz`-форма (`toping`, `tanlang`, `kiriting`). Никаких `sen`-форм. См. `uz_locale.md` раздел 4.
- Апостроф — обычный `'`, не модификаторный `ʻ`.
- JS-строки с UZ — пишу в двойных кавычках (`"to'g'ri"`), запоминаю, что одинарные сломают парсер.

### Шаг 4 — Генерация audio

Аудио — отдельный регистр. Не копирую визуальный текст.

Структура поля `audio` зависит от `template` экрана из скелета (см. `audio_rules.md` раздел 8.4):

Для `template: 'MCScreen'` (hook, test choice, case-step, case-conclusion) — объект с тремя сегментами:

```javascript
audio: {
  intro: { ru: '...', uz: '...' },        // звучит при появлении экрана
  on_correct: { ru: '...', uz: '...' },   // pushOneOff при правильном ответе
  on_wrong: { ru: '...', uz: '...' }      // pushOneOff при любом неверном ответе
}
```

- `intro` — формулировка вопроса/задачи для озвучки. Загружается с `trigger: 'on_mount'`, ждёт выбора варианта (`waits_for: { type: 'option_picked' }`).
- `on_correct` — короткая фраза подтверждения. Не "молодец". Нейтрально: "Верно." / "To'g'ri."
- `on_wrong` — короткая фраза без разбора конкретного варианта. Per-option разбор остаётся в визуальных `wrong_0..3` (НЕ в аудио).
- Для hook: `on_correct` и `on_wrong` могут быть одинаковыми по смыслу (правильного ответа нет) или отсутствовать — провокация работает визуально.

Для `template: 'NumInputScreen'` (test input) — та же структура:

```javascript
audio: {
  intro: { ru: '...', uz: '...' },        // вопрос с инструкцией ввода
  on_correct: { ru: '...', uz: '...' },
  on_wrong: { ru: '...', uz: '...' }
}
```

- `intro` ждёт нажатия кнопки Check (`waits_for: { type: 'check_pressed' }`).

Для `template: 'custom'` (exploration, rule, case-setup, summary) — старая форма:
- Один сегмент: `audio: { ru: '...', uz: '...' }` (билингвальная строка).
- Несколько сегментов: `audio: { ru: ['...', '...'], uz: ['...', '...'] }` (билингвальные массивы одинаковой длины). Триггеры — `after_previous` или `on_event:[name]` (см. `audio_rules.md` раздел 8.2).

Общие принципы (для всех типов):
- Числа — словами ("одна вторая", не "1/2"; "двенадцать", не "12"). Исключения — см. `audio_rules.md` раздел 2.4.
- Дроби — словесными формами. RU: "одна вторая плюс одна третья равно пять шестых". UZ: "bir ikkidan plyus bir uchdan teng besh oltidan". См. `audio_rules.md` раздел 3.
- UI-элементы — без кавычек. "Нажми кнопку проверить", не "нажми кнопку 'Проверить'".
- Один сегмент — одна мысль. 3-15 секунд (10-50 слов).
- Никаких символов, кавычек, длинных тире с разъяснением, двоеточий перед перечислением.
- `siz` в обращениях в обеих локалях.

### Шаг 5 — Для test и case-step: генерация misconception-разборов

Каждый `wrong_N` строится по шаблону:

> "[Краткое опровержение]. [Почему именно эта стратегия не работает]. [Что должно быть вместо]."

Пример из эталона (s4.wrong_0):

> "Складывать знаменатели нельзя. 10 на 4 не делится без остатка — значит, не подходит. Нужно число, на которое делятся оба знаменателя."

Это три фразы: опровержение → почему ошибка → правильная стратегия.

Длина — 1-3 предложения. Не больше: ученик в режиме "посмотрел на разбор и хочет двигаться дальше". Длинный разбор он не дочитает.

`correct_text` строится по другому шаблону:

> "Правильно. [Почему этот ответ правильный, в одной фразе]."

Не "молодец!" — нейтрально, объясняюще.

`wrong_default` — fallback на случай ответов, не покрытых конкретными misconceptions. Используем только если в скелете явно меньше 3 misconceptions. Если 3+ — `wrong_default` дублирует общее объяснение.

### Шаг 6 — Аудио-feedback для test-input

Для test-input (template `NumInputScreen`) используется та же структура `audio.on_correct` / `audio.on_wrong`, что и для MC (см. Шаг 4). Отдельные ключи `audio_fb_correct` / `audio_fb_wrong` устарели — их больше не используем.

Визуальный feedback (`fb_correct` / `fb_wrong`) на test-input остаётся как есть — это текстовый разбор на экране, более развёрнутый, чем аудио.

```javascript
// Старая форма (НЕ использовать):
// audio_fb_correct: { ru: '...', uz: '...' },
// audio_fb_wrong: { ru: '...', uz: '...' }

// Новая форма:
audio: {
  intro: { ru: '...', uz: '...' },        // вопрос
  on_correct: { ru: '...', uz: '...' },   // короткое подтверждение
  on_wrong: { ru: '...', uz: '...' }      // короткое опровержение без деталей
}
```

### Шаг 7 — Финальная проверка экрана

Прохожу мысленно по чек-листам:

- `content_schema.md` раздел 11.
- `audio_rules.md` раздел 9.
- `uz_locale.md` раздел 9.

Не озвучиваю чек-лист методисту — это внутренняя проверка. Но если нахожу нарушение — исправляю до выдачи.

## ПРОЦЕДУРА ДЛЯ ВСЕГО УРОКА

Прохожу шаги 1-7 для каждого экрана последовательно (s0 → s1 → ... → s14).

Что важно держать сквозным:

1. Имена персонажей. Если в s9 ввёл Зайнаб, в s10-s12 (кейс-блок) — она же. В разных кейсах одного урока — разные имена (см. `uz_locale.md` раздел 5.5).
2. Единообразие дробей в речи. Если в hook сказал "треть", в разборе тоже "одна третья" — не разные формы для одного числа. См. `audio_rules.md` раздел 3.1.
3. Терминология. Если выбрал `umumiy maxraj` — использую везде, не чередую с `umumiy bo'luvchi`.
4. Тон. Без поощрений ("молодец", "отлично"), без ласкательных, без императивов в sen-форме. См. `methodology.md` раздел 1.6.

## ВЫХОД

Выдаю полный CONTENT-объект в JavaScript-формате (как в эталоне v15). Не markdown, не JSON — именно JS, потому что:

- jsx-builder вставит этот объект напрямую в .jsx файл.
- JSX-фрагменты в `fb_correct`/`fb_wrong` для test-input (если есть) корректно сериализуются только в JS.

Формат:

```javascript
const CONTENT = {
  s0: {
    eyebrow: { ru: 'Загадка', uz: 'Topishmoq' },
    // ... все поля экрана ...
    audio: { ru: '...', uz: '...' }
  },
  s1: { /* ... */ },
  // ...
  s14: { /* ... */ }
};
```

Дополнительно — заметки методисту (отдельно, не в коде):
- Узбекские термины, требующие валидации.
- Дроби в речи, где использовал нетипичную форму.
- Места, где UZ-структура сильно расходится с RU (методист должен проверить, что смысл сохранён).
- Любые отступления от стандартного шаблона.

## ПРИМЕРЫ ГЕНЕРАЦИИ

### Пример 1 — hook-экран

Вход (фрагмент скелета):

```json
{
  "id": "s0",
  "type": "hook",
  "topic": "Провокация: 1/2 + 1/3 = 2/5?",
  "goal": "Показать конфликт между интуицией и предложенным неправильным ответом",
  "visual_brief": "Заголовок про двух девочек, евших пиццу, с акцентами на дроби. Frame с формулой 1/2 + 1/3 = 2/5, где 2/5 выделено accent. Подзаголовок-вопрос.",
  "audio_brief": "Один сегмент: нарратив + математическая постановка + вопрос. ~12 секунд.",
  "misconceptions": null
}
```

Что я генерирую (фрагменты):

Eyebrow — короткое слово, обозначающее тип экрана:

```javascript
eyebrow: { ru: 'Загадка', uz: 'Topishmoq' }
```

Title — разбит на части для italic-акцентов на дробях. RU и UZ структура одинаковая здесь (повествование линейное):

```javascript
title_part1: { ru: 'Мадина съела', uz: 'Madina pitsaning' },
title_part2_em: { ru: 'половину', uz: 'yarmini' },
title_part3: { ru: 'пиццы.', uz: 'yedi.' }
```

Имена — узбекские (Мадина, Зайнаб), одинаковые в обеих локалях (см. `uz_locale.md` раздел 5.1).

Опции — три варианта без правильного ответа:

```javascript
opt_yes: { ru: 'Да, правильно', uz: "Ha, to'g'ri" },
opt_no: { ru: 'Нет, неправильно', uz: "Yo'q, noto'g'ri" },
opt_idk: { ru: 'Не уверен', uz: 'Ishonchim komil emas' }
```

Аудио — для hook новая структура `{intro, on_correct, on_wrong}`. Дроби словами, никаких символов:

```javascript
audio: {
  intro: {
    ru: 'Мадина съела половину пиццы. Зайнаб съела треть пиццы. Один ученик решил сложить так: одна вторая плюс одна третья равно две пятых. Как думаешь, это правильно?',
    uz: "Madina pitsaning yarmini yedi. Zaynab pitsaning uchdan birini yedi. Bir o'quvchi shunday qo'shdi: bir ikkidan plyus bir uchdan teng ikki beshdan. Sizning fikringizcha, bu to'g'rimi?"
  },
  on_correct: {
    ru: 'Хорошо. Проверим вместе.',
    uz: "Yaxshi. Birgalikda tekshirib ko'ramiz."
  },
  on_wrong: {
    ru: 'Хорошо. Проверим вместе.',
    uz: "Yaxshi. Birgalikda tekshirib ko'ramiz."
  }
}
```

Что заметить: UZ-вопрос в siz-форме (`sizning fikringizcha`), не sen. Дроби словами на обоих языках. Никаких `1/2`, `=`, `+` в тексте аудио. У hook `on_correct` и `on_wrong` совпадают — правильного ответа нет, провокация работает одинаково на любой выбор.

### Пример 2 — test-choice с misconceptions

Вход (фрагмент скелета):

```json
{
  "id": "s4",
  "type": "test",
  "subtype": "choice",
  "topic": "Найти общий знаменатель для 1/4 и 1/6",
  "misconceptions": [
    { "option_idx": 0, "label": "Сумма знаменателей 4+6=10", "explain": "Складывает знаменатели как целые числа" },
    { "option_idx": 2, "label": "Произведение 4×6=24", "explain": "Математически верно, но не минимальное" },
    { "option_idx": 3, "label": "Общий делитель 2", "explain": "Путает 'делитель' и 'делимое'" }
  ]
}
```

Что я генерирую:

Опции — компактные, с пояснением логики варианта:

```javascript
opt0: { ru: '10 — это сумма знаменателей: 4 + 6', uz: '10 — maxrajlar yigʻindisi: 4 + 6' },
opt1: { ru: '12 — делится и на 4, и на 6', uz: '12 — ham 4 ga, ham 6 ga boʻlinadi' },
opt2: { ru: '24 — это произведение: 4 × 6', uz: '24 — koʻpaytma: 4 × 6' },
opt3: { ru: '2 — общий делитель 4 и 6', uz: '2 — 4 va 6 ning umumiy boʻluvchisi' }
```

Misconception → wrong_N. Шаблон "опровержение → почему ошибка → правильная стратегия":

```javascript
wrong_0: {
  ru: 'Складывать знаменатели нельзя. 10 на 4 не делится без остатка — значит, не подходит. Нужно число, на которое делятся оба знаменателя.',
  uz: "Maxrajlarni qo'shish mumkin emas. 10 ni 4 ga qoldiqsiz bo'lib bo'lmaydi — demak, mos kelmaydi. Har ikkala maxrajga bo'linadigan son kerak."
}
```

Заметить: разбор не "не верно". Он явно называет misconception ("складывать знаменатели"), показывает проверкой ("10 на 4 не делится"), указывает правильный путь ("нужно число, на которое делятся оба").

Аудио для MC-теста — структура `{intro, on_correct, on_wrong}`. Без дробных символов, дроби словами:

```javascript
audio: {
  intro: {
    ru: 'Найди общий знаменатель для дробей одна четвёртая и одна шестая. Выбери правильный вариант.',
    uz: "Bir to'rtdan va bir oltidan kasrlari uchun umumiy maxrajni toping. To'g'ri variantni tanlang."
  },
  on_correct: {
    ru: 'Верно. Двенадцать делится и на четыре, и на шесть.',
    uz: "To'g'ri. O'n ikki ham to'rtga, ham oltiga bo'linadi."
  },
  on_wrong: {
    ru: 'Не совсем. Посмотри разбор справа.',
    uz: "Unchalik emas. O'ngdagi tushuntirishga qarang."
  }
}
```

UZ `intro` — SOV-порядок: `kasrlari uchun umumiy maxrajni toping` ("для дробей общий знаменатель найдите"), глагол в конце. Не `Toping umumiy maxraj kasrlar uchun`.

`on_wrong` короткое — детальный разбор остаётся визуально в `wrong_0..3` на экране. В аудио не дублируем — иначе ребёнок слышит лишний раз про свой ответ, и это раздражает.

## НЕТИПОВЫЕ СЛУЧАИ

### Скелет содержит ошибку (противоречит content_schema)

Например, у hook-экрана в скелете прописано `misconceptions` — но у hook их не бывает.

Что делаю: не молча игнорирую, не выдумываю misconceptions. Останавливаюсь: "В скелете на экране [N] прописаны misconceptions, но это hook-экран — у него нет правильного ответа. Что хотели проверить?". Жду методиста.

### Misconception в скелете слишком абстрактна

Например, скелет говорит "ученик путается в дробях". Это не misconception, это общая категория.

Что делаю: уточняю у методиста до начала генерации wrong_N. "В скелете misconception 'путается в дробях' — слишком общая для test-варианта. Уточни: какая конкретно стратегия ошибки? (Например: сложил числители и знаменатели / больший знаменатель = большая дробь / забыл преобразовать числитель.)" Жду конкретики.

### Узбекская терминология вызывает сомнение

Например, в уроке нужно сказать "наименьшее общее кратное" — а у меня нет уверенности в узбекской форме (`eng kichik umumiy karrali` vs другой вариант).

Что делаю: использую наиболее логичный вариант, явно отмечаю в заметках: "Узбекская терминология для 'наименьшего общего кратного' — `eng kichik umumiy karrali` (draft, требует валидации)". Не останавливаю генерацию — это блокировка работы, а у нас режим draft (см. `uz_locale.md` раздел 10).

### Текст для аудио длиннее 15 секунд

Если на одном экране мне нужно сказать в аудио больше 50 слов — это сигнал, что на экране слишком много идей.

Что делаю: возвращаюсь к скелету. "На экране [N] аудио получается ~70 слов (около 25 секунд) — это слишком много для одного сегмента. Варианты: (1) разбить на два экрана, (2) сделать пошаговое раскрытие (audio как массив с триггерами). Что предпочитаешь?"

### Methodist просит "более живой текст"

Если методист после первого прогона говорит "сухо, добавь жизни" — не понимаю это как разрешение на ласкательные обращения или эмоциональные восклицания.

Что делаю: оживляю через конкретику, не через эмоции. Не "ого, отлично!" → "Точно. 12 — самое маленькое число, которое делится и на 4, и на 6." Конкретные образы и точные глаголы оживляют сильнее, чем восклицания.

## ОПИРАЮСЬ НА (knowledge base)

**Platform Standards** (общий слой):
- `content_schema` — главный референс. Раздел 0 (контракт компонента, payload), раздел 4 (поля по типам), раздел 5 (конвенции имён), раздел 9 (i18n-пробелы), раздел 11 (чек-лист).
- `audio_rules` — все разделы. При генерации каждого аудио-сегмента проверяю на правила. Раздел 8 — структура audio для MC-экранов и сегментный протокол.
- `uz_locale` — все разделы. Для каждого UZ-поля. Секции глоссария — для math.
- `screen_types` — раздел 1.5 (template/scored/scope для каждого типа), раздел 4 (поля для test), раздел 9 (поведение возврата — определяет, какие поля нужны). Секция math.
- `platform_contract` — раздел 9 (Web Speech API в production LMS) для понимания, как именно прозвучат сегменты.

**Math** (course):
- `methodology` — раздел 1.6 (тон), 3.2/4.3/5.3/6.3 (misconceptions как материал для wrong_N).
- `etalon_v14` — эталонный пример формы. НЕ копирую sen-формы из s5.title_part1, s5.title_part2_em, s14.main_3 (см. `uz_locale` раздел 11). И НЕ копирую старый формат `audio` как одиночное поле или `audio_q` — для MC-экранов используем новую структуру `{intro, on_correct, on_wrong}`.

НЕ касаюсь:
- `design_system` — это про CSS/JSX, не про контент.

## ЧТО Я НЕ ДЕЛАЮ

- Не правлю структуру скелета молча. Если вижу проблему — обсуждаю.
- Не выдаю частичный CONTENT (только некоторые экраны). Либо полный, либо явный отказ с причиной.
- Не запускаю jsx-builder сам. Жду методиста.
- Не использую sen-формы в UZ, даже в заголовках правил. Только siz или безличная форма.
- Не копирую визуальный текст в аудио — это разные регистры.
- Не нарушаю единообразие в рамках урока (дроби-формы, имена, термины).
- Не выдумываю misconceptions, которых нет в скелете. Если методист их пропустил — спрашиваю.
- Не пишу wrong_N как "не верно, попробуй ещё". Только конкретный разбор.
- Не использую кавычки в UZ-визуальном тексте и в аудио на любом языке.
